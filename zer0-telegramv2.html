Telegram v2 — Conversational Trading

Bring the in-app trading flow to Telegram. User asks zer0 to analyze a market in chat, then says "buy me X shares" — zer0 routes the trade to Polymarket and triggers a wallet signature on the user's phone.

SCOPE: design NEXT FEATURE DEPENDS ON: client-side submit (just landed)

1. What we're building

An end-to-end conversational trading flow inside Telegram, with these capabilities:

  • Connect. User runs /connect in the zer0 Telegram bot. Bot generates a WalletConnect v2 pairing URI; user opens it via deep link into MetaMask Mobile (or any WalletConnect-compatible wallet). Session is persisted server-side, keyed by Telegram user ID.
  • Ask. User: "what do you think about the Spencer Pratt LA mayor market?" Bot fetches the market via Polymarket Gamma, runs the same superforecaster analysis the in-app brain-tick uses, replies with the rationale and current edge.
  • Trade. User: "buy me $10 of YES" or "sell 20 shares at market". Bot parses the intent via Groq/OpenAI, builds the EIP-712 typed-data payload, sends it via the WalletConnect session to the user's wallet. User signs in MetaMask Mobile. Bot routes the signed order to Polymarket's CLOB.
  • Confirm. Bot replies with the outcome — order ID, fill price, on-chain tx link — same surface the in-app RecentTradesBubble shows today.

Why this matters for the demo

The in-app web flow proves "agent → signed order → on-chain fill". The Telegram flow proves a more compelling story: natural language → autonomous agent reasoning → on-chain fill, from your phone, anywhere. It's the one-line pitch the demo needs.

3. WalletConnect v2 — the central piece

The crux is enabling the Telegram bot to request signatures from the user's mobile wallet. WalletConnect v2 is the standard protocol for this. Two relevant SDKs:

  • @reown/walletkit — newer, Reown-maintained successor to @walletconnect/web3wallet. Bot-side acts as a dApp client requesting signatures from wallets.
  • @reown/appkit — higher-level, mostly for browser apps. We want the lower-level Sign API client (@walletconnect/sign-client or its v2.x successor).

Flow at the protocol level

  1. Bot calls signClient.connect({ requiredNamespaces: { eip155: { chains: ['eip155:137'], methods: ['eth_signTypedData_v4', 'eth_sendTransaction', 'personal_sign'], events: ['chainChanged','accountsChanged'] } } }). Returns a uri.
  2. Bot sends the URI to the user as a Telegram message — both as a tappable deep-link (https://metamask.app.link/wc?uri=…) and a QR code image. MetaMask Mobile (or any WC2 wallet) opens it.
  3. User approves the pairing in their wallet. Bot receives session_connect event with the user's EOA address.
  4. Bot persists (telegram_user_id → session_topic, eoa_address) in supabase.
  5. When the user wants to trade, bot calls signClient.request({ topic, chainId: 'eip155:137', request: { method: 'eth_signTypedData_v4', params: [eoa, typedDataJson] } }).
  6. Wallet popup appears on the user's phone. Approve → bot receives the signature.
  7. Bot constructs the signed order and posts it to Polymarket.

Pairing UX in Telegram

Telegram doesn't render QR codes natively, but it does render images. The bot can:

  1. Send the deep link as plain text (https://metamask.app.link/wc?uri=wc:...) — tappable on mobile, opens MetaMask Mobile directly.
  2. Generate a QR code server-side (qrcode npm package) and send as a photo for desktop Telegram users.
  3. Include a "Copy URI" inline button for power users.

Why this is the right protocol

WalletConnect v2 is wallet-agnostic. The user doesn't need MetaMask specifically — Rainbow, Trust Wallet, Coinbase Wallet, etc. all support WC2. The bot stays decoupled from any particular wallet vendor. It also handles session refresh, multi-chain, and disconnect events out of the box.

4. Intent parsing — natural language to trade

The user says things like:

  • "buy 10 shares of YES on Spencer Pratt LA mayor"
  • "sell $5 of NO at market on the Iran market"
  • "what's my position on the FIFA World Cup market?"
  • "close my Bitcoin $150k position"

We use Groq's llama-3.1-8b-instant (already in deps, already used by the classifier in brain-tick) with structured output to parse this into:

{
  "intent": "open_trade" | "close_trade" | "analyze_market" | "status",
  "market_query": "Spencer Pratt LA mayor" | null,
  "side": "BUY" | "SELL" | null,
  "size_kind": "shares" | "usd" | null,
  "size_value": 10 | 5 | null,
  "outcome": "YES" | "NO" | null,
  "confidence": 0.0-1.0
}

If confidence < 0.7 or required fields are missing, bot asks a clarifying question instead of executing. Important guardrail: always echo the intent back to the user before signing. Bot replies:

"OK — buying ~10 YES shares of Spencer Pratt LA mayor at current ask of $0.014 (≈$0.14 total). Approving in your wallet now."

Then the wallet popup. The user has a final visual confirmation in the wallet itself (MakerAmount/TakerAmount), which is the actual security guarantee — the bot can lie about what it's signing but the wallet won't.

5. Trade execution paths — where the bot lives

The bot still needs to post the signed order to Polymarket. That's the same POST that gets geoblocked on Vercel iad1. Three options:

A. Bot on a non-blocked host

If the Telegram bot runs on Oracle Cloud, Fly.io, Railway, etc. in an allowed region (Ireland, Brazil, Sweden, etc.), it can post directly. The relay key pattern (RELAY_PRIVATE_KEY for API auth) works as-is.

Cleanest. The bot is server-side anyway — no client-side complexity to worry about.

Comments in src/lib/polymarket/clob.ts already mention "Oracle Cloud workers" — that may be where the bot is intended to run.

B. Bot uses user's own API keys

Mirror the browser pattern: derive a Polymarket API key per user from their EOA signature (which we'd request once via WalletConnect during /connect). Post on their behalf using their HMAC creds. User's IP isn't relevant because the bot's IP is what hits Polymarket — same problem as A.

Useful for accountability / per-user rate limits, but doesn't solve geoblock. Skip unless we also need the per-user auth for other reasons.

C. Bot delegates to a web view

Bot opens a Telegram WebApp (mini-app) for the user to confirm; the WebApp runs the same client-side flow we just built. User signs in WebApp's wallet context, post happens from the browser.

Breaks the "in chat" narrative — the user has to leave Telegram chat. Worse demo.

Recommendation: A. The bot already needs a server — provision it in Dublin (Fly.io free tier, or Oracle ARM VM) and we're done.

6. Schema additions

A few new supabase columns/tables to support per-user WalletConnect sessions:

create table public.tg_wc_sessions (
  telegram_user_id bigint primary key,
  session_topic text not null,
  eoa_address text not null,
  expires_at timestamptz not null,
  created_at timestamptz default now(),
  last_used_at timestamptz default now()
);

-- (optional) per-user telegram→eoa map for /api/trade/list lookups
create table public.tg_user_wallets (
  telegram_user_id bigint primary key,
  primary_eoa text not null,
  created_at timestamptz default now()
);

The existing trades table already has user_address, so per-user trade history works without modification — bot just inserts trades with the user's EOA address and the existing RecentTradesBubble logic (if we ever surface this in the web app) would pick it up automatically.

8. Open questions to resolve before building

  1. Where does the bot already run? If it's on Oracle Cloud, what region? If it's a Vercel function, it's geoblocked the same way the web app is — we'll need to move it.
  2. Should the bot derive its own API keys per Telegram user (Option B above), or use the relay's single API key (Option A)? The relay key is simpler; per-user keys give us better audit trails and per-user rate limit isolation on Polymarket's side. Lean simple for v1.
  3. Session expiry policy. WC2 sessions have a max lifetime (typically 7 days). Renewal: silent re-pairing, or require /connect again? Silent renewal is nicer but adds complexity.
  4. What does "buy at market" mean precisely? Our in-app flow uses FAK (Fill-And-Kill) at the cross-the-spread price. Same for Telegram. But if the user says "sell at $0.20" we should honor that as a true limit — needs intent schema to express both modes.
  5. How do we surface fills back into Telegram? Push the order ID + Polygonscan link as a Telegram message? Or also reflect in the web app's RecentTradesBubble (we already do this via the existing notify endpoint)? Probably both.
  6. Multi-wallet per Telegram user. Power users may want to use different wallets for different trades. v1: one EOA per Telegram user, set on /connect. Re-running /connect replaces it.

9. Security & trust considerations

  • The wallet popup is the actual source of truth. No matter what the bot tells the user, the typed-data shown in MetaMask Mobile (MakerAmount, TakerAmount, Side, TokenId) is what the user actually signs. As long as the user reads the popup, the bot cannot steal funds.
  • Bot-side validation. Before sending a typed-data sign request via WC, the bot must verify: the recommendation/market exists, the size is within the user's stated intent, the price is within a sane bound of the current order book. Avoids the bot accidentally requesting a 10x-larger trade than the user said.
  • Session hijack mitigation. WC sessions are bound to the pairing keys derived during the initial handshake. Anyone with the session topic alone cannot send signature requests. Still, session storage on the bot's DB is sensitive — treat it like an auth token. Encrypt at rest if Supabase doesn't already.
  • Rate limiting. Reuse the existing src/lib/trades/rate-limit.ts with a 'tg-trade' scope to cap trades per Telegram user per minute.
  • Spoofed Telegram messages. Grammy verifies updates against the bot token, so we can trust the from.id field. The bot's session lookup keys on that ID.

11. Why not just use Polymarket's own UI?

Polymarket has a web UI and Telegram trading via PolyTrader and similar third-party bots. We're not competing with the trading execution — we're offering analysis + execution in one conversational surface. zer0 has its own opinion. The trade is the consequence of the conversation, not the start of it.

The differentiator is in the analyze step (Tetlock-style superforecasting) and the autonomous brain-tick that surfaces opportunities. The Telegram surface is just the most natural place to consume that.
