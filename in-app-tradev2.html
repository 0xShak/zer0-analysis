In-app trade execution — handoff v2

CURRENT STATUS: BROKEN (decimals) ERROR: invalid amounts SCOPE: one rounding bug remains; the rest of V2 works

One-line summary

The V1→V2 migration (in-app-tradev1.html), the deposit-wallet flow, the ERC-7739 POLY_1271 signature wrapping, and the HMAC builder authentication are all working end-to-end. The user can now: connect → wrap USDC.e into the deposit wallet → relayer-batch approve pUSD → sign the order → submit. The pipeline reaches Polymarket's matcher cleanly. The matcher rejects with: 

submit failed: clob_rejected: invalid amounts, the market buy orders
maker amount supports a max accuracy of 2 decimals,
taker amount a max of 4 decimals

This is a rounding-precision mismatch in buildTypedData. We're using the SDK's limit-order rounding rules, but submitting as FAK — which Polymarket validates against market-order decimal limits. The fix is small (constrained to one function in src/lib/polymarket/clob.ts). The previous engineer (Opus 4.7) said: "I think this one should be easy."

0. Read this first

Everything in §3 is load-bearing — don't undo it

This codebase went through three large arcs since v1: full V2 SDK migration, the deposit-wallet flow, and Polymarket builder authentication. Every piece is what makes the order signature actually reach Polymarket's matcher. The remaining bug is the matcher saying "I got your order, but the numbers don't fit my decimal rules." Treat the existing architecture as ground truth — fix the rounding in one place, don't refactor around it.

In particular: do not revert to the V1 SDK (Polymarket V2 retired V1 entirely on 2026-04-28), do not revert to direct EOA trading (rejected with "maker address not allowed, use the deposit wallet flow"), do not remove the ERC-7739 wrap suffix from order signatures (deposit-wallet isValidSignature needs it to parse the signature).

a. V1 → V2 SDK migration (commit: V2 migration round)

Replaced @polymarket/clob-client@5.8.1 with @polymarket/clob-client-v2@1.0.6. Swapped exchange addresses to V2:

  • CTF Exchange V2: 0xE111180000d2663C0091e4f400237545B87B996B
  • NegRisk CTF Exchange V2: 0xe2222d279d744050d28e00520010520000310F59

EIP-712 domain version bumped from "1" to "2"; name still "Polymarket CTF Exchange".

Tried using the V2 SDK's ExchangeOrderBuilderV2.buildOrderTypedData(order) but the SDK's package.json only exports . (no subpath imports — TS resolves @polymarket/clob-client-v2/dist/... as "Cannot find module"). Mints fresh pUSD inside the wallet on every wrap call.

d. Deposit-wallet flow (the big one)

Polymarket V2 server-side rejects every EOA-maker order with clob_rejected: maker address not allowed, please use the deposit wallet flow. Universal — confirmed via py-clob-client-v2 issues #51/#52/#53, the Firefly tweet, no documented allowlist. The deposit wallet is the V2 trading identity.

Each user gets a per-EOA ERC-1967 minimal proxy, CREATE2-deterministic from (factory, bytes32(owner)):

  • DepositWalletFactory: 0x00000000000Fb5C9ADea0298D729A0CB3823Cc07 (role-gated, only Polymarket's relayer can call deploy)
  • DepositWalletImplementation: 0x58CA52ebe0DadfdF531Cde7062e76746de4Db1eB (current; the factory supports setImplementation for future upgrades)

Built src/lib/polymarket/deposit-wallet.ts:

  • deriveDepositWalletAddress(eoa) — pure CREATE2 via the SDK's deriveDepositWallet helper.
  • isDepositWalletDeployed(eoa) — multi-RPC getCode probe.
  • getBrowserRelayClient — instantiates @polymarket/builder-relayer-client's RelayClient with the ethers v5 JsonRpcSigner over window.ethereum.
  • deployDepositWalletFromBrowser — POST WALLET_CREATE to relayer-v2.polymarket.com. Sponsored; no user signature.
  • executeDepositWalletBatchFromBrowser — signs a Batch EIP-712 payload (domain "DepositWallet" v "1", verifyingContract = wallet) authorizing the relayer to execute calls FROM inside the wallet. We use this for the one-time pUSD.approve(V2Exchange) and (SELL only) CTF.setApprovalForAll(V2Exchange) — both bundled in a single signature.
  • encodePusdApproveCall / encodeCtfApproveAllCall — call-data builders for the batch.

Polymarket relayer URL: https://relayer-v2.polymarket.com/

e. POLY_1271 ERC-7739 nested-TypedDataSign signature

Deposit-wallet orders use signatureType = 3 (POLY_1271). The signature isn't a raw EIP-712 sig — it's an ERC-7739 wrap:

signature = innerSig(65 bytes)
          || appDomainSep(32 bytes)
          || contentsHash(32 bytes)
          || V2_ORDER_TYPE_STRING bytes (186 bytes)
          || lenHex(2 bytes, big-endian uint16 of typeString length)

where:

  • innerSig = eth_signTypedData_v4(typedData) where typedData has primaryType: 'TypedDataSign', outer EIP-712 domain = V2 EXCHANGE domain ("Polymarket CTF Exchange" / "2" / chain 137 / V2 exchange contract), and message contains { contents: orderMessage, name: 'DepositWallet', version: '1', chainId: 137, verifyingContract: depositWallet, salt: bytes32(0) }.
  • appDomainSep = keccak256(EIP712Domain || nameHash || versionHash || chainId || verifyingContract) — the exchange domain separator.
  • contentsHash = keccak256(ORDER_TYPEHASH || all order struct fields).
  • The deposit wallet's isValidSignature uses the typeString suffix to parse out the inner ECDSA, recover the EOA owner, and confirm the order was signed by the wallet's controlling key.

All of this is in src/lib/polymarket/clob.ts (buildTypedData + buildWrapSuffix + computeAppDomainSep + computeOrderContentsHash) and src/components/TradeCard.tsx (the concat '0x' + innerSig.slice(2) + wrapSuffix.slice(2)).

Verified working: the matcher accepts the signature (which proves the wrap is correct); only the amounts fail validation.

f. Polymarket HMAC builder authentication (relayer auth)

The first attempt at the deposit-wallet flow hit POST relayer-v2.polymarket.com/submit → 401 { error: 'invalid authorization' }. Polymarket's relayer requires builder authentication on every /submit call — even for WALLET_CREATE.

The product owner registered as a builder at polymarket.com/settings?tab=builder → Create new → View API Details. That dialog shows the HMAC trio once with the warning: "Never share your key or put it in client-side code."

Wired the official @polymarket/builder-signing-sdk in REMOTE mode:

  • Pinned to ^0.0.8 (must match builder-relayer-client's peer; 1.0.0 caused two divergent BuilderConfig classes in node_modules and TS rejected the cross-version pass).
  • POLYMARKET_BUILDER_API_KEY, POLYMARKET_BUILDER_SECRET, POLYMARKET_BUILDER_PASSPHRASE in .env.local and Vercel — server-only, never NEXT_PUBLIC_.
  • NEXT_PUBLIC_POLYMARKET_BUILDER_CODE (the bytes32 attribution code) is public — gets stamped into every V2 order's builder field.
  • src/app/api/polymarket/builder-sign/route.ts takes POST { method, path, body, timestamp? } and returns { POLY_BUILDER_API_KEY, POLY_BUILDER_TIMESTAMP, POLY_BUILDER_PASSPHRASE, POLY_BUILDER_SIGNATURE } using buildHmacSignature(secret, ts, method, path, body). Rate-limited per IP.
  • Browser BuilderConfig is REMOTE-mode pointing at ${window.location.origin}/api/polymarket/builder-sign — the SDK calls our route before every relayer + CLOB write, attaches the returned headers to the outbound request.

The secret never reaches the browser. The api-key + passphrase do (the SDK sends them as headers), but Polymarket's auth design treats the secret as the only credential that matters.

g. Balance sync after fund/approve

Polymarket's matcher caches per-account balances. After wrapping pUSD into the deposit wallet and approving from inside it, the cache is stale and the matcher would reject orders with insufficient-balance. syncDepositWalletBalance(client) in clob-browser.ts calls client.updateBalanceAllowance({ asset_type: 'COLLATERAL' }) right before order submission. Best-effort — if it fails we log and continue.

4. The current bug — invalid amounts (decimals)

What we see

After the order signs and gets POSTed, Polymarket's response body is:

{ success: false, errorMsg: 'invalid amounts, the market buy orders maker amount supports a max accuracy of 2 decimals, taker amount a max of 4 decimals' }

The TradeCard maps it into { kind: 'rejected', reason: errorMsg } and renders the red text. The trade row in trades gets status='rejected' + failure_reason populated.

What this error means

Polymarket's V2 matcher enforces per-side, per-order-type decimal limits on the raw maker/taker amounts. For market BUY orders specifically (FAK / FOK), the rule is:

  • makerAmount (in pUSD, what the buyer pays) — at most 2 decimal places of precision. i.e. cents. 4.94 ✓, 4.9405 ✗.
  • takerAmount (in shares, what the buyer receives) — at most 4 decimal places. 12.0500 ✓.

These limits are different from the limit-order rules (limit orders use (size: tick-dependent, amount: tick-dependent) from ROUNDING_CONFIG). The V2 SDK has separate helpers for the two paths:

  • node_modules/@polymarket/clob-client-v2/dist/order-builder/helpers/buildOrderCreationArgs.js — limit orders (GTC, GTD)
  • node_modules/@polymarket/clob-client-v2/dist/order-builder/helpers/buildMarketOrderCreationArgs.js — market orders (FAK, FOK)
  • node_modules/@polymarket/clob-client-v2/dist/order-builder/helpers/getMarketOrderRawAmounts.js — the rounding helper for market amounts

Our inlined buildTypedData uses the limit path's rounding even though we submit as OrderType.FAK.

Where the suspect code is

In src/lib/polymarket/clob.ts, getOrderRawAmounts (lines ~91–119). BUY branch:

if (side === 'BUY') {
  const rawTakerAmt = roundDown(size, cfg.size);            // shares, 2 decimals (cfg.size=2)
  let rawMakerAmt = rawTakerAmt * rawPrice;                  // USD value
  if (decimalPlaces(rawMakerAmt) > cfg.amount) {             // cfg.amount=4 for tickSize "0.01"
    rawMakerAmt = roundUp(rawMakerAmt, cfg.amount + 4);
    if (decimalPlaces(rawMakerAmt) > cfg.amount) {
      rawMakerAmt = roundDown(rawMakerAmt, cfg.amount);      // 4 decimals ← TOO MANY for market BUY
    }
  }
  return { rawMakerAmt, rawTakerAmt };
}

For the reproduction trade (size=$5, price=$0.41, tickSize=0.01):

  • rawTakerAmt = roundDown(5/0.41, 2) = roundDown(12.1951…, 2) = 12.19. (Wait — sizeShares is computed in prepare/route.ts at line 160 as sizeUsd / price, then passed to buildTypedData as size. So size=12.1951, then roundDown to 2 dec = 12.19. The "12.05 shares" in the UI must use a slightly different price — possibly the recommendation price vs the book-crossing price.)
  • rawMakerAmt = 12.19 * 0.41 = 4.9979 (or similar). 4 decimals.
  • parseUnits('4.9979', 6) = 4_997_900.
  • The matcher reads this as $4.9979 — 4 decimals — and rejects.

For a market BUY, we want rawMakerAmt snapped to 2 decimals: 4.99 (round down) → 4_990_000.

Where to look for the canonical rounding

Read node_modules/@polymarket/clob-client-v2/dist/order-builder/helpers/getMarketOrderRawAmounts.js and buildMarketOrderCreationArgs.js. They're the SDK's own implementation of market-order rounding. Compare them to getOrderRawAmounts.js (limit) — the difference between the two files is the fix.

5. Most likely root cause + fix sketch (ranked)

  1. Limit-order rounding applied to a market order. Highest confidence. The SDK explicitly distinguishes the two paths; our inline reimplementation only ports the limit path. Fix: either (a) add a market-order branch in getOrderRawAmounts (or a sibling helper) that rounds maker/taker per market rules, OR (b) directly import the SDK's buildMarketOrderCreationArgs — but the package's exports field blocks subpath imports, so probably inline again.

    Verify: read the two SDK files, diff them, port the difference. Probably ~10 lines.

  2. Per-market tick-size mismatch. The market's actual tickSize might not be "0.01" — could be "0.001" or "0.0001". If we hard-default to "0.01" but the market is finer, the SDK's ROUNDING_CONFIG yields different (size, amount) precision. Less likely because the matcher's error names a fixed 2/4-decimal rule, not "matches the tick", but worth confirming the tickSize value coming out of getMarketMeta for the test market.

    Verify: log tickSize in prepare/route.ts right before buildTypedData.

  3. BUY vs SELL semantic swap. In market BUY, maker = pUSD, taker = shares. In market SELL, the swap would flip (maker = shares, taker = pUSD). Our error refers to BUY rules. The SELL branch (currently untested end-to-end) might have the same shape issue mirrored. Out of scope for this bug but worth a flag.
  4. Off-by-one in roundUp/roundDown helpers. Our roundUp and roundDown use Math.ceil(n * f) / f / Math.floor(n * f) / f — float precision can introduce a phantom 5th decimal. If the matcher sees 4.9978999999999998 the JSON serializer might emit a long string. Lower likelihood given the SDK uses the same Math idiom.

  5. Card shows: submit failed: clob_rejected: invalid amounts, … in red.

8. Things that work today (don't break these)

  • Wallet connect via injected EIP-1193 provider.
  • Chain auto-switch + add-chain fallback.
  • Allowance preflight: pUSD/USDC.e/CTF reads scoped to the right address (wallet vs EOA), multi-RPC fallback.
  • Deposit-wallet deterministic address derivation (matches the SDK's deriveDepositWallet; verified an arbitrary EOA returns the correct on-chain wallet address).
  • Deposit-wallet deployment via Polymarket relayer (sponsored gas, no user signature, WALLET_CREATE).
  • USDC.e → pUSD wrap with to = depositWallet.
  • Relayer-batched pUSD.approve(V2Exchange) signed from inside the deposit wallet (one EIP-712 Batch signature).
  • Polymarket HMAC builder authentication — secret stays server-side; per-request signing; SDK in REMOTE mode.
  • ClobClient V2 with signatureType: POLY_1271, funderAddress: depositWallet, builderConfig: { builderCode }.
  • API-key derivation cached in localStorage['zer0:pm-creds:<eoa>'].
  • ERC-7739 wrap suffix construction (buildWrapSuffix in clob.ts) — the matcher accepts the signature, proving the wrap math is correct.
  • Balance sync (updateBalanceAllowance) — the matcher sees the freshly-wrapped pUSD.
  • brain-tick recommendations every 30 minutes.
  • RecentTradesBubble per-user feed.
  • force-recommendation CLI.
