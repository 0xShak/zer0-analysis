0. Read this first

Don't undo anything in §3 without reading why

This codebase went through a long debug arc. Every piece in §3 ("what we built") is load-bearing and fixes a specific previous failure mode. The current bug is downstream of all of it. Treat the existing architecture as ground truth and dig into why Polymarket calls our signed order "version mismatched" rather than reverting things.

In particular: do not revert the client-side submit to a server-side submit (you'll hit the geoblock from §3.f), do not remove the allowance preflight (first-time users will get silent rejections), do not change OrderType.FAK back to GTC (we'd lose instant-fill semantics and create phantom resting orders).

3. What we built today (in chronological order)

You'll see commits going back through these — the latest commits do the most-recent work. This is the journey, including dead ends, because some of the "fixes" inform why the architecture is shaped the way it is.

a. Wired up wallet signing (TradeCard.tsx)

The frontend was deliberately stubbed: throw new Error('wallet signing wired up on day 5'); at TradeCard.tsx:45 (now removed). The backend's /api/trade/prepare + /api/trade/submit were complete but the client never signed or submitted.

Added eth_signTypedData_v4, normalised the signed order shape to match SignedOrderShape (zod) in src/lib/trades/validators.ts, and posted to the server submit route.

b. Chain-switch logic (Polygon 0x89)

MetaMask was on Ethereum mainnet (chainId 1); the EIP-712 domain we signed was for chainId 137. MetaMask refused the sign with RPC error "Provided chainId '137' must match the active chainId '1'". Added wallet_switchEthereumChain call before signing, with wallet_addEthereumChain fallback if the user didn't have Polygon configured (error code 4902).

c. EIP-1193 error rendering

The error footer initially showed [object Object]. EIP-1193 errors from injected wallets are plain { code, message } objects, not Error instances — so String(e) in the catch block produced the useless stringification. Fixed by extracting obj.message and special-casing code 4001 ("signature rejected"), 4100 ("wallet not authorized"). Lives in TradeCard's catch block today.

d. USD vs shares unit bug (subtle and important)

The analyzer's JSON schema returns suggested_size_usd, the DB column is size_usd — but the Polymarket SDK's buildOrderCreationArgs treats size as shares, not USD. The original prepare route passed sizeUsd as size directly, meaning a "$100 trade" on a $0.25 market became a 100-share order = $25 of actual exposure.

Fixed in src/app/api/trade/prepare/route.ts: const sizeShares = sizeUsd / price;, then pass size: sizeShares to buildTypedData. The trades.size_usd column still stores the user's stated USD intent.

e. Custom size input + min-order-size guard

TradeCard got a $-prefixed numeric input (range $1-$100, matching the server's PrepareBody cap in src/lib/trades/validators.ts). Also shows the share-equivalent estimate so the user can sanity-check what'll appear in the MetaMask popup.

Server-side: getBookContext() in clob.ts returns min_order_size from Polymarket's order book. Prepare gates with sizeShares < minOrderSize → 400 below_min_size + a suggestedMinUsd field. Polymarket markets vary — some need ≥ 5 shares, some ≥ 10, etc.

f. Geoblock discovery (and the Vercel Hobby plan trap)

Trades initially returned "Trading restricted in your region" from Polymarket. The block is IP-based and applies to the Vercel function's egress, not the user's browser. Polymarket blocks 33 countries on POST /order (US, UK, Germany, France, Italy, Netherlands, Australia, Belgium, Russia, Iran, plus more — see docs.polymarket.com/developers/CLOB/geoblock). Vercel Hobby pins Node functions to iad1 (Washington DC, US).

We tried export const preferredRegion = 'dub1'; — Vercel Hobby silently ignored it. Confirmed via x-vercel-id header which still showed :iad1: as the function region after deploy. (Pro plan would honor it, but the user chose not to upgrade.)

Decision: move order submission to the browser. The user's IP is what gets checked, not ours. SDK is browser-safe (browser-or-node detection, WebCrypto for HMAC, ~80KB, no Node-only imports). This is the architecture you see today.

g. Allowance preflight + approve flow

Polymarket needs USDC.e approve(CTF_EXCHANGE, MAX_UINT256) from the user's EOA before any BUY can settle. Without it, the matcher silently kills the order (no transferFrom permission). For SELL orders, additionally needs CTF.setApprovalForAll(CTF_EXCHANGE, true).

Built: server-side allowance reader (src/lib/polymarket/allowance.ts using viem + multi-RPC fallback), /api/trade/allowance route, client-side approve transactions (src/lib/web3/approve.ts), TradeCard integration that prompts approval before signing if needed.

Note: the allowance route returns { unknown: true } (not 503) when all RPCs fail, so the UI shows a soft warning instead of silently bypassing the approval step. There used to be a race condition where TradeCard would let the user click execute before preflight finished; fixed by re-running preflight inline at the start of execute() if approval state is null.

h. FAK semantics + cross-the-spread pricing

Originally used OrderType.GTC (Good Til Cancel), which meant orders that didn't fill at submit time would rest in Polymarket's book indefinitely — bad UX because the user thinks they "executed" but nothing happens on-chain. Switched to OrderType.FAK (Fill And Kill): match what's possible at the order price, kill the unfilled remainder.

For FAK to actually fill, the order price must be at or beyond the opposite side of the book (BUY at the ask, SELL at the bid). Added getBestExecutionPrice(tokenId, side) in clob.ts that reads the order book and returns the cross-the-spread price. Prepare overrides the recommendation's price with this book price.

Concretely: model recommends SELL @ $0.25, but current best bid is $0.24 — we sign at $0.24 so the order actually matches against the bids, instead of resting at $0.25 forever.

i. Post-submit polling (the matcher is async)

Polymarket's postOrder response is just an acknowledgment. Matching + on-chain settlement happen async (1-6 seconds later). The initial response is often { success: true, orderID: '...' } with no status, no makingAmount, no transactionsHashes. We were misclassifying these as "cancelled" instantly.

Added pollOrderResolution (server-side in clob.ts) and pollOrderFromBrowser (browser-side in clob-browser.ts) that poll getOrder(orderID) every 1s for up to 8s. Read the resulting status field (LIVE → MATCHED / CANCELED / UNMATCHED / KILLED) and size_matched to determine the true outcome.

j. SDK error-response shape gotcha

Polymarket SDK's http-helpers/index.js#errorHandling rewraps axios 4xx errors into { error: '...', status: 4xx }, NOT the documented { success: false, errorMsg } shape. Our soft-rejection check was only catching success === false, so HTTP-level errors fell through to the success path, leading to phantom-cancelled rows with empty status data.

Fixed in src/components/TradeCard.tsx by also checking for the error field: const httpErrorMsg = typeof resultObj.error === 'string' ? resultObj.error : null; if (httpErrorMsg || isSoftReject) { … }. This is what finally surfaced the geoblock error string verbatim (and is now surfacing order_version_missmatch the same way).

k. Privacy fix (RecentTradesBubble)

Submit used to insert into thoughts with scope='app', which is globally visible via RLS (thoughts_read_app policy in supabase/migrations/0001_initial.sql:177-178). Every user could see every other user's trade activity in the ThoughtsBubble. Removed the thoughts insert.

Built src/components/app/RecentTradesBubble.tsx reading from /api/trade/list?address=… which filters by user_address. Per-user activity, no cross-user leak.

l. Client-side submission (the big refactor)

Created src/lib/polymarket/clob-browser.ts with:

  • getOrCreatePolymarketClient(ethereum, address) — builds ethers v5 Web3Provider over window.ethereum, derives ClobClient. One-time per EOA, calls createOrDeriveApiKey() (one MetaMask EIP-712 popup), caches HMAC creds in localStorage['zer0:pm-creds:<eoa-lowercased>'].
  • submitOrderFromBrowser(client, signedOrder) — calls client.postOrder(signedOrder, OrderType.FAK) from the user's browser.
  • pollOrderFromBrowser(client, orderID) — same polling logic as the server-side version.

Replaced the /api/trade/submit server route with /api/trade/notify — pure DB write, no Polymarket calls. The deleted /api/trade/submit route is gone from src/app/api/trade/ entirely.

4. The current bug — order_version_missmatch

What we see

After the user signs an order, the browser ClobClient posts to Polymarket and gets back something like:

{ error: 'order_version_missmatch', status: 4xx }

or possibly

{ success: false, errorMsg: 'order_version_missmatch' }

(I'm not 100% sure which shape — the Vercel notify logs will tell you. Search for [trade/notify] rejected in Vercel logs and the reason field will have the exact response Polymarket gave us.)

What this error means

Polymarket's CTF Exchange contract verifies the EIP-712 signature against a domain separator built from { name, version, chainId, verifyingContract }. If any of those four fields drift between what the SDK encodes when building the typed-data and what the contract expects when verifying, the recovered signer doesn't match the order's signer field and the contract reverts. Polymarket's CLOB API does a pre-flight version check before forwarding to the chain — that pre-flight is what's emitting order_version_missmatch.

The "version" field is the most likely drift. The "name" and "verifyingContract" are pretty stable; chainId is locked to 137. But version can be bumped when Polymarket redeploys / upgrades the Exchange contract logic, and an older SDK won't know about the new version string.

Where the suspect domain is constructed

In src/lib/polymarket/clob.ts:154-181, buildTypedData instantiates an ExchangeOrderBuilder:

const builder = new ExchangeOrderBuilder(
  exchangeAddress,  // CTF_EXCHANGE or NEG_RISK_CTF_EXCHANGE based on negRisk
  CHAIN_ID,         // 137
  getTypedDataStubSigner() as any,
);
…
return builder.buildOrderTypedData(order);

The ExchangeOrderBuilder lives in node_modules/@polymarket/clob-client/dist/order-utils/exchange.order.builder.js. Read it. It hardcodes the EIP-712 domain — including the version string. Compare what it hardcodes to what's currently deployed on Polygon at:

  • CTF Exchange: 0x4bFb41d5B3570DeFd03C39a9A4D8dE6Bd8B8982E
  • NegRisk Exchange: 0xC5d563A36AE78145C45a50134d48A1215220f80a

Look at the contract's eip712Domain() view function (or DOMAIN_SEPARATOR()) on Polygonscan. That's the ground truth. Whatever the SDK encodes must match exactly.

5. Most likely root causes (ranked)

  1. SDK version drift. We're on @polymarket/clob-client@5.8.1 (see package.json). Polymarket has published newer versions since. If they bumped the Exchange contract's EIP-712 version string in a redeploy and updated the SDK accordingly, our 5.8.1 still signs the old version. First thing to check.

    Verify: npm view @polymarket/clob-client versions --json | tail -20. If there's a 6.x or even 5.9+, that's almost certainly the fix. Bump, test, ship.

  2. Wrong exchange contract selection. Our exchangeForMarket(negRisk) picks between CTF Exchange and NegRisk Exchange based on the recommendation's neg_risk field, which comes from market.negRisk via Polymarket's Gamma API. If the Gamma API's negRisk flag is wrong (stale or absent for some markets), we'd sign against the wrong exchange's domain. Polymarket would then complain because the signature claims to be for the wrong contract.

    Verify: log the negRisk value used in prepare; cross-check against polymarket.com's UI for the same market. The US-Iran market we've been testing has negRisk: false (per our force-rec output) — confirm that's still true.

  3. Different EIP-712 version field per exchange variant. Even if SDK and contract addresses match, the CTF Exchange and NegRisk Exchange might have different EIP-712 version strings deployed. If ExchangeOrderBuilder uses a single hardcoded version for both, one of them would mismatch. Less likely than #1 but possible.
  4. Order field tampering. The order struct we sign (salt, nonce, expiration, feeRateBps, signatureType, etc.) must hash to the same struct hash the contract expects. If we're sending integer types where the contract expects strings or vice-versa, the hash differs. buildTypedData already uses the SDK's helper to construct args, so this is unlikely to be wrong — but worth eyeballing the order object in buildTypedData after construction.
  5. Signature payload corruption in the browser path. The browser does const signedOrder = { ...typedData.message, signature }; then passes to client.postOrder. If the SDK's orderToJson serializer transforms field types differently than how we re-stringify them, the on-the-wire JSON might not match what was hashed. Long shot, but inspectable by logging the exact request body that client.postOrder sends.

  • An EVM wallet (MetaMask, Coinbase Wallet, etc.) on Polygon.
  • USDC.e balance: not native USDC. Polymarket uses the bridged variant at 0x2791Bca1f2de4661ED88A30C99A7a9449Aa84174, not Circle's native USDC at 0x3c499c542cEF5E3811e1192ce70d8cC03d5c3359. The user can swap via QuickSwap or Polygon Portal.

  • The user does NOT need to go through polymarket.com's "Enable Trading" wizard — that wizard is for the POLY_GNOSIS_SAFE proxy flow (signatureType=2). We use signatureType=0 (EOA direct).
